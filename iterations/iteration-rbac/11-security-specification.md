## 1. Threat Model

| ID | Threat | Attack Vector | Control | Residual Risk |
|----|--------|--------------|---------|---------------|
| T-01 | Credential stuffing | Automated login attempts with leaked email/password combos | Rate limit: 5 req/60s per IP on `/api/auth/login` | Attacker rotates IPs. Mitigated by monitoring + future MFA. |
| T-02 | Brute force password | Repeated login attempts against one account | Same rate limit + bcrypt cost 12 (~250ms per hash) | Slow hash limits throughput. Account lockout in Phase 2. |
| T-03 | Session hijacking | Steal access token from browser | httpOnly + Secure + SameSite=Lax cookies. Token not accessible via JS. | Physical device access. Mitigated by short token lifetime (15 min). |
| T-04 | Refresh token replay | Stolen refresh token used after legitimate user refreshes | Token rotation: old token revoked on each refresh. Revoked token triggers security alert. | If attacker uses token before legitimate user. Short window. |
| T-05 | JWT forgery | Attacker crafts JWT with elevated role | HS256 signature with server-side secret. Secret not exposed in client code. | Secret compromise. Mitigate: rotate secret, use RS256 in Phase 3. |
| T-06 | Privilege escalation | Student modifies request to access admin endpoints | Server-side role check on every endpoint via `require_role` / `require_permissions` dependency. | Misconfigured endpoint missing guard. Mitigate: integration tests verify every endpoint. |
| T-07 | Horizontal privilege escalation | Student A accesses Student B's data | `require_self_access` middleware verifies `request.user.id == target_id`. | Endpoint missing self-access check. Mitigate: test suite covers cross-student access. |
| T-08 | Faculty data leakage | Faculty accesses unassigned section | `require_section_access` checks `faculty_section_assignments` table on every request. | DB inconsistency. Mitigate: assignment deletion immediately revokes access. |
| T-09 | SQL injection | Malicious input in login email, search, or filter params | SQLAlchemy parameterized queries. No raw SQL. Pydantic input validation. | ORM bypass. Extremely unlikely with SQLAlchemy. |
| T-10 | XSS via stored data | Malicious script in user name or department name | React auto-escapes JSX output. No `dangerouslySetInnerHTML`. Backend stores as-is; frontend escapes on render. | Third-party component rendering raw HTML. Code review mitigates. |
| T-11 | CSRF | Cross-site request triggers state-changing actions | SameSite=Lax cookies block cross-origin POST. CORS restricts allowed origins. | SameSite=Lax allows GET from external links (but GET endpoints are read-only). |
| T-12 | Password hash exposure | API response accidentally includes `password_hash` | Pydantic response models explicitly exclude `password_hash`. Integration test TC-SEC-007 verifies. | New endpoint forgetting to use response model. Mitigate: code review + test. |
| T-13 | Token in URL/logs | JWT accidentally logged or included in URL query params | Tokens stored only in httpOnly cookies, never in URL params or request body (except login). Logging middleware excludes cookie values. | Misconfigured reverse proxy logging full headers. |
| T-14 | Inactive user access | Deactivated user continues to access API | JWT validation loads user from DB and checks `is_active`. Deactivation revokes all refresh tokens. | Access token valid for up to 15 min after deactivation. Acceptable risk for Phase 1. |

## 2. Authentication Controls

### 2.1 Password Hashing

| Property | Value | Rationale |
|----------|-------|-----------|
| Algorithm | bcrypt | Industry standard, adaptive cost factor |
| Cost factor | 12 | ~250ms on modern hardware. Balances security and UX. |
| Library | passlib[bcrypt] | Well-maintained Python bcrypt wrapper |
| Salt | Automatically generated by bcrypt (embedded in hash) | Each hash unique even for same password |

**Implementation:**
```python
from passlib.context import CryptContext
pwd_context = CryptContext(schemes=["bcrypt"], bcrypt__rounds=12)
```

### 2.2 Token Storage

| Token | Storage Location | httpOnly | Secure | SameSite | Max-Age |
|-------|-----------------|----------|--------|----------|---------|
| Access token (JWT) | Cookie: `access_token` | Yes | Yes (prod) | Lax | 900s (15 min) |
| Refresh token | Cookie: `refresh_token` | Yes | Yes (prod) | Lax | 604800s (7 days) |

**Why cookies over localStorage:**
- httpOnly prevents XSS from accessing tokens.
- Secure flag prevents transmission over HTTP.
- SameSite=Lax prevents CSRF on state-changing requests.
- Cookies are automatically sent with every request — no client-side token management.

**What is NOT done:**
- Tokens are NEVER stored in localStorage.
- Tokens are NEVER stored in sessionStorage.
- Tokens are NEVER included in URL query parameters.
- Tokens are NEVER returned in response body (only set via Set-Cookie header).

### 2.3 Token Lifetimes

| Token | Lifetime | Rationale |
|-------|----------|-----------|
| Access token | 15 minutes | Short-lived to limit damage if compromised. Frequent refresh is transparent to user. |
| Refresh token | 7 days | Long enough for "remember me" behavior. User stays logged in for a week without re-entering credentials. |

### 2.4 Refresh Token Rotation

On every `POST /api/auth/refresh`:
1. Old refresh token is immediately revoked (`revoked_at = now()`).
2. New refresh token is generated and stored (SHA-256 hash in DB).
3. New access token is issued.

**Why rotation:**
- If an attacker steals a refresh token and the legitimate user also refreshes, the stolen token becomes invalid.
- Detects token theft: if a revoked token is presented, it indicates the token was used elsewhere.

### 2.5 JWT Payload

```json
{
  "sub": "user-uuid",
  "email": "admin@cec.edu.in",
  "role": "admin",
  "iat": 1706400000,
  "exp": 1706400900
}
```

**What is NOT in the JWT:**
- `password_hash` — never.
- Department or section IDs — looked up from DB on each request for freshness.
- Permissions list — derived from role at request time to avoid stale permissions.

## 3. Authorization Controls

### 3.1 Server-Side Enforcement

Every endpoint is protected by FastAPI dependencies that run BEFORE the route handler:

```python
# Example: three layers of protection
@router.get("/api/sections/{section_id}/attendance")
async def get_attendance(
    section_id: str,
    current_user: User = Depends(get_current_user),      # Layer 1: Authentication
    _role: User = Depends(require_permissions(             # Layer 2: Permission check
        Permission.ATTENDANCE_VIEW)),
    _scope: User = Depends(require_section_access),        # Layer 3: Department scoping
):
    ...
```

**No client-side enforcement:** Frontend route guards are for UX only. They prevent users from seeing pages they can't use. The actual security enforcement is always server-side.

### 3.2 Permission Derivation

Permissions are derived from the user's role using a static map defined in `app/core/permissions.py`:

```python
ROLE_PERMISSIONS = {
    "admin": [all permissions],
    "faculty": [attendance, marks, announcements, timetable view, student view],
    "student": [attendance view (self), marks view (self), fee view (self), announcements, timetable],
}
```

**Why static map over database-stored permissions:**
- Simpler implementation for Phase 1.
- 3 roles with fixed permissions — no need for dynamic permission assignment yet.
- Easier to test (unit test the map directly).
- Future: if dynamic permissions are needed (Phase 2+ with HOD/Principal), migrate to DB-stored permissions.

### 3.3 Resolution Precedence

```
1. Authentication (JWT valid?) ------> 401 if failed
2. Role permission check ------------> 403 if role lacks permission
3. Admin bypass (skip scope check) --> Allow
4. Faculty section assignment check -> 403 if not assigned
5. Student self-access check --------> 403 if accessing other student
6. All passed -----------------------> Allow
```

### 3.4 Admin Bypass

`admin` role bypasses all department/section scope checks. This means:
- Admin can view any student's data.
- Admin can access any section's attendance.
- Admin can manage all departments and users.

This is by design — admin is the college administrative staff who needs institution-wide access.

## 4. Credential Endpoint Hardening

### Before RBAC

| Scenario | Behavior |
|----------|----------|
| Any API call | No authentication required |
| Student data | Accessible to anyone with the URL |
| Fee records | No access control |
| Attendance | No access control |

### After RBAC

| Scenario | Behavior |
|----------|----------|
| Any API call without cookie | 401 Unauthorized |
| Student data by faculty | Only if assigned to student's section |
| Student data by student | Only own records |
| Fee records | Admin: all; Student: own only; Faculty: none |
| User management | Admin only |
| Department/section management | Admin only |

### Password Hash Masking

`password_hash` is **never** returned in any API response:

1. **Pydantic response models** explicitly define which fields are returned — `password_hash` is not included.
2. **Integration test TC-SEC-007** verifies this across all user-returning endpoints.
3. **Code review checklist** includes checking for `password_hash` leakage.

## 5. Rate Limiting

### Login Endpoint

| Property | Value |
|----------|-------|
| Endpoint | `POST /api/auth/login` |
| Limit | 5 requests per 60 seconds |
| Key | Client IP address (`X-Forwarded-For` if behind proxy) |
| Response when exceeded | `429 Too Many Requests` |
| Library | slowapi (wraps limits) |

### Implementation

```python
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

@router.post("/login")
@limiter.limit("5/minute")
async def login(request: Request, ...):
    ...
```

### What Is NOT Rate-Limited (Phase 1)

- Token refresh (`POST /api/auth/refresh`) — frequent automatic calls.
- Other API endpoints — authenticated users are trusted for Phase 1.
- Future: per-user rate limiting on write operations.

## 6. Input Validation

### Pydantic Validation

All request bodies are validated by Pydantic models before reaching route handlers:

```python
class LoginRequest(BaseModel):
    email: EmailStr        # Validates email format
    password: str = Field(min_length=1)

class CreateUserRequest(BaseModel):
    email: EmailStr
    password: str = Field(min_length=8, max_length=128)
    name: str = Field(min_length=2, max_length=100)
    role: Literal["admin", "faculty", "student"]
    usn: Optional[str] = Field(None, pattern=r"^[0-9][A-Z]{2}[0-9]{2}[A-Z]{2,3}[0-9]{3}$")
```

Invalid input returns `422 Unprocessable Entity` with detailed error messages.

### SQL Injection Prevention

- **SQLAlchemy ORM** uses parameterized queries for all database operations.
- **No raw SQL** is used anywhere in the codebase.
- **Pydantic validation** rejects malformed input before it reaches the ORM.

### XSS Prevention

- **React** auto-escapes all JSX interpolation (`{variable}` in templates).
- **No `dangerouslySetInnerHTML`** is used.
- **Content-Type headers** are always `application/json` — no HTML rendering from API responses.

## 7. CORS Configuration

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.CORS_ORIGINS.split(","),  # e.g., ["http://localhost:5173"]
    allow_credentials=True,     # Required for cookie-based auth
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### Rules

| Environment | Allowed Origins |
|------------|-----------------|
| Development | `http://localhost:5173` (Vite dev server) |
| Production | `https://cec-student.example.com` (exact domain) |

**`allow_credentials=True` is required** because the frontend sends cookies with every request. Without this, browsers block cookie transmission on cross-origin requests.

**`allow_origins` must NOT be `["*"]`** when `allow_credentials=True` — this is blocked by the CORS spec. Always use explicit origins.

## 8. What RBAC Does Not Cover

### 8.1 Network-Level Security

RBAC operates at the application layer. It does not address:
- TLS/SSL termination (handled by reverse proxy / hosting platform).
- DDoS protection (handled by CDN / WAF).
- Firewall rules (handled by cloud provider / infrastructure).

### 8.2 Database Encryption

SQLite stores data unencrypted on disk. For Phase 1 (development), this is acceptable. For production:
- Consider SQLite encryption extensions (SQLCipher) or migrate to PostgreSQL with encrypted storage.
- Ensure the server's filesystem has appropriate access controls.

### 8.3 Email Verification

Phase 1 does not verify email ownership. Admin creates accounts with assumed-valid emails. Future phases may add:
- Email verification on account creation.
- Password reset via email.

### 8.4 Multi-Factor Authentication

Not in Phase 1. Future consideration for admin accounts:
- TOTP-based 2FA.
- SMS verification.

### 8.5 API Key Authentication

No API keys are issued. All access is via JWT cookies from browser sessions. Service-to-service authentication (if needed) is a future concern.

### 8.6 Data Encryption in Transit

Assumes HTTPS in production. Development uses HTTP on localhost. The `Secure` cookie flag is set based on environment configuration.

## 9. Audit Trail — Phase 3 Specification

### 9.1 Audit Log Table

```python
class AuditLog(Base):
    __tablename__ = "audit_logs"

    id = Column(String, primary_key=True, default=lambda: str(uuid.uuid4()))
    actor_id = Column(String, ForeignKey("users.id"), nullable=True)  # Null for system actions
    actor_email = Column(String, nullable=True)
    actor_role = Column(String, nullable=True)
    action = Column(String, nullable=False)  # e.g., "user.login", "attendance.mark"
    resource_type = Column(String, nullable=True)  # e.g., "user", "attendance"
    resource_id = Column(String, nullable=True)
    metadata_json = Column(String, nullable=True)  # JSON-serialized extra data
    ip_address = Column(String, nullable=True)
    timestamp = Column(String, nullable=False, default=lambda: datetime.now(timezone.utc).isoformat())
```

### 9.2 Audit Service

```python
class AuditService:
    def __init__(self, db: Session):
        self.db = db

    async def log(
        self,
        actor: Optional[User],
        action: str,
        resource_type: Optional[str] = None,
        resource_id: Optional[str] = None,
        metadata: Optional[dict] = None,
        ip_address: Optional[str] = None,
    ):
        entry = AuditLog(
            actor_id=actor.id if actor else None,
            actor_email=actor.email if actor else None,
            actor_role=actor.role if actor else None,
            action=action,
            resource_type=resource_type,
            resource_id=resource_id,
            metadata_json=json.dumps(metadata) if metadata else None,
            ip_address=ip_address,
        )
        self.db.add(entry)
        self.db.commit()
```

### 9.3 Actions to Instrument

| Action | When | Metadata |
|--------|------|----------|
| `auth.login.success` | Successful login | `{ ip }` |
| `auth.login.failure` | Failed login attempt | `{ email, ip, reason }` |
| `auth.logout` | User logs out | `{ ip }` |
| `user.create` | Admin creates user | `{ new_user_id, role }` |
| `user.update` | Admin updates user | `{ user_id, changed_fields }` |
| `user.deactivate` | Admin deactivates user | `{ user_id }` |
| `user.password_reset` | Admin resets password | `{ user_id }` |
| `user.bulk_import` | Admin bulk imports | `{ imported, skipped, errors }` |
| `department.create` | Admin creates dept | `{ dept_id, code }` |
| `section.create` | Admin creates section | `{ section_id }` |
| `assignment.create` | Faculty assigned | `{ faculty_id, section_id }` |
| `assignment.delete` | Faculty unassigned | `{ faculty_id, section_id }` |
| `attendance.mark` | Faculty marks attendance | `{ section_id, date, student_count }` |
| `marks.enter` | Faculty enters marks | `{ section_id, exam_type }` |

### 9.4 Retention Policy

- Audit logs retained for **3 years** (AICTE/NAAC compliance).
- Logs older than 3 years archived to cold storage.
- Admin can export audit logs as CSV via `GET /api/admin/audit-logs?export=csv`.

## 10. Security Sign-off Checklist

Before declaring RBAC iteration complete, verify:

### Authentication

- [ ] All endpoints except `/api/auth/login` and `/api/health` return 401 without a valid JWT cookie.
- [ ] Passwords are hashed with bcrypt cost factor 12.
- [ ] `password_hash` is absent from every API response (verified by TC-SEC-007).
- [ ] Access token expires in 15 minutes.
- [ ] Refresh token expires in 7 days.
- [ ] Refresh token rotation works: old token revoked, new token issued.
- [ ] Logout revokes refresh token and clears cookies.
- [ ] Deactivated user cannot log in and their tokens are revoked.

### Authorization

- [ ] Admin can access all endpoints.
- [ ] Faculty cannot access admin-only endpoints (user management, department management).
- [ ] Student cannot access faculty or admin endpoints.
- [ ] Faculty cannot access sections they are not assigned to.
- [ ] Student cannot access another student's records.
- [ ] Admin bypasses all scope checks.

### Rate Limiting

- [ ] Login endpoint blocks after 5 attempts in 60 seconds (429).
- [ ] Rate limit resets after the window expires.

### Input Validation

- [ ] All request bodies are validated by Pydantic.
- [ ] SQL injection attempts in login, search, and filter params do not cause errors.
- [ ] Invalid email format is rejected at DTO level.
- [ ] USN format is validated for student creation.

### CORS

- [ ] `allow_credentials=True` is set.
- [ ] `allow_origins` is an explicit list (not `*`).
- [ ] Requests from unauthorized origins are rejected.

### Cookies

- [ ] `httpOnly=True` on both cookies.
- [ ] `Secure=True` in production.
- [ ] `SameSite=Lax` on both cookies.

### Tests

- [ ] All unit tests pass (TC-UNIT-*).
- [ ] All integration tests pass (TC-INT-*).
- [ ] All security tests pass (TC-SEC-*).
- [ ] Coverage >= 80% on auth modules.

### Feature Flag

- [ ] `AUTH_ENABLED=false` bypasses all auth checks (verified by TC-EDGE-015).
- [ ] `AUTH_ENABLED=true` enforces all auth checks.
